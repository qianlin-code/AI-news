# AI掘金头条新闻系统 — 后端设计说明文档

基于 FastAPI + SQLAlchemy async + Redis 构建的新闻系统后端，支持用户注册登录、新闻浏览、收藏和历史记录等功能。

## 1. 技术栈

| 分类 | 技术 | 版本 |
|------|------|------|
| 后端框架 | FastAPI | 0.139 |
| ASGI 服务器 | Uvicorn | 0.49 |
| 数据库 | MySQL (via aiomysql) | 8+ |
| ORM | SQLAlchemy (async) | 2.0.51 |
| 缓存 | Redis (redis-py async) | 8.0 |
| 密码加密 | passlib + bcrypt | 1.7.4 / 3.2.2 |
| 配置管理 | pydantic-settings + .env | 2.14 |
| 数据验证 | Pydantic v2 | 2.13 |

## 3. 配置管理

使用 `.env` + `pydantic-settings` 统一管理所有配置项：

```ini
# .env 示例
DB_HOST=localhost
DB_PORT=3306
DB_USER=root
DB_PASSWORD=1234
DB_NAME=news_app
DB_ECHO=true              # 开发环境打印 SQL，生产设 false

REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_DB=0

DEBUG=true                # 控制异常详情是否返回客户端
CORS_ORIGINS=http://127.0.0.1:5173,http://localhost:5173
```

代码中通过 `config/settings.py` 的 `Settings` 类统一读取，提供 `settings.database_url` 等计算属性。所有硬编码值已消除。

## 4. 数据库设计

所有模型共享 `models/__init__.py` 中的统一 `Base`，通过 `Base.metadata.create_all()` 建表。

| 表名 | ORM 类 | 说明 |
|------|--------|------|
| `user` | User | 用户信息（用户名唯一、密码 bcrypt 加密） |
| `user_token` | UserToken | 认证令牌（UUID4，7 天过期，upsert 更新） |
| `news_category` | Category | 新闻分类（name 唯一） |
| `news` | News | 新闻（FK→news_category，索引: category_id + publish_time） |
| `favorite` | Favorite | 收藏（UniqueConstraint: user_id + news_id） |
| `history` | History | 浏览历史（去重+更新 view_time 策略） |

## 5. 缓存设计（Cache-Aside 模式）

```
请求 → 查 Redis → 命中? → 返回
                   ↓ 未命中
              查 MySQL → 写 Redis → 返回
```

| 缓存 Key | 过期时间 | 用途 |
|----------|----------|------|
| `news:categories` | 7200s (2h) | 分类列表（最稳定） |
| `news:list:{category_id}:{page}:{size}` | 1800s (30min) | 新闻列表 |
| `news:detail:{news_id}` | 1800s (30min) | 新闻详情 |
| `news:related:{news_id}` | 120s (2min) | 相关推荐（变化快） |

**序列化策略：** ORM → Pydantic `model_dump(mode="json")` → dict → `json.dumps()` → Redis。`mode="json"` 自动将 `datetime` 转为 ISO 字符串，解决 JSON 序列化报错问题。

**过期时间设计原则：** 数据越稳定，过期时间越长；避免所有 Key 同时过期引起缓存雪崩。

**启动预热：** 服务启动时通过 `lifespan` 事件自动将分类列表和前 3 个分类的首页新闻加载到 Redis，避免首个用户触发缓存冷启动（所有请求穿透到 MySQL）。预热失败不影响正常使用。

## 6. 日志系统

- **控制台**：`DEBUG` 级别（开发环境显示所有日志）
- **`logs/app.log`**：`INFO` 级别，10MB 轮转，保留 7 天
- **`logs/error.log`**：`ERROR` 级别，单独记录便于排查
- **请求日志中间件**：每个 HTTP 请求自动记录 `GET /api/news/detail?id=1 → 200 (42.3ms)`
- 第三方库日志降噪：`aiomysql`/`sqlalchemy.engine` 设为 WARNING

## 7. 认证机制

- 用户登录后生成 UUID4 Token，7 天过期
- Token 存储在 `user_token` 表，支持 upsert（同一用户多次登录只保留最新 Token）
- 客户端在 `Authorization` 请求头传入 Token（兼容 `Bearer xxx` 和 `xxx` 两种格式）
- `get_current_user` 依赖注入：校验 Token 有效性 → 返回 `User` ORM 对象

## 8. 异常处理

全局注册 5 个异常处理器（具体在前，抽象在后）：

| 异常类型 | HTTP 状态码 | 说明 |
|----------|-------------|------|
| `RequestValidationError` | 422 | Pydantic 参数校验失败，返回第一条错误信息 |
| `HTTPException` | 按异常 | 业务层主动抛出的异常，透传状态码和详情 |
| `IntegrityError` | 400 | 数据库约束冲突（用户名重复、外键不存在等） |
| `SQLAlchemyError` | 500 | 通用数据库异常 |
| `Exception` | 500 | 全局兜底 |

所有异常都会通过 logger 记录；开发模式 (`DEBUG=true`) 返回堆栈和路径，生产模式只返回提示语。

## 9. 统一响应格式

所有接口通过 `utils/response.py` 的 `success_response()` 统一返回：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

## 10. 性能优化

- **连接池**：DB pool_size=10 + max_overflow=20
- **异步全链路**：FastAPI async + SQLAlchemy async + aiomysql + redis.asyncio
- **数据库索引**：category_id、publish_time、username、phone、token、user_id+news_id 联合唯一
- **Redis 缓存**：Cache-Aside 模式减少 DB 查询
- **RESP2 协议**：兼容 Redis 5.x（Windows 环境）
